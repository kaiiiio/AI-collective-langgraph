Workflow pressure

The hard part is not calling the model

The hard part is controlling what happens before and after the model: data loading, prompting, parsing, validation, retries, human approval, saving, monitoring, and debugging.

LangGraph structures the workflow. LangSmith shows what happened inside it.

LangGraph + LangSmith

LangGraph makes the steps explicit.

Core idea

LangGraph is workflow control for AI systems

State
  -> Node
  -> State update
  -> Edge chooses next node
  -> Repeat until END
Outputsame state object, with new fields added after each node
What it meansLangGraph is controlled state movement. Every node receives state and returns the part it changed.
Every slide after this asks one question: what changed in state?

Complete starter

A full tiny LangGraph app

from typing import TypedDict
from langgraph.graph import StateGraph, START, END

class DemoState(TypedDict, total=False):
    user_message: str
    response: str

def answer_node(state: DemoState) -> dict:
    # Read state, do one job, return only the update.
    return {"response": f"Received: {state['user_message']}"}

graph = StateGraph(DemoState)
graph.add_node("answer", answer_node)
graph.add_edge(START, "answer")
graph.add_edge("answer", END)

app = graph.compile()
print(app.invoke({"user_message": "hello"}))
Output{"user_message": "hello", "response": "Received: hello"}
What it meansThe graph did not replace normal Python. It ran one function, merged the returned field into state, and ended with a larger state object.

Connected functions

Multi-node graphs are still simple

from typing import TypedDict
from langgraph.graph import StateGraph, START, END

class PageState(TypedDict, total=False):
    company_id: str
    context: str
    prompt: str
    raw_response: str

def fetch_context(state: PageState) -> dict:
    return {"context": f"Context for {state['company_id']}"}

def build_prompt(state: PageState) -> dict:
    return {"prompt": f"Write page using {state['context']}"}

def call_model(state: PageState) -> dict:
    return {"raw_response": "{...model json...}"}

graph = StateGraph(PageState)
graph.add_node("fetch_context", fetch_context)
graph.add_node("build_prompt", build_prompt)
graph.add_node("call_model", call_model)
graph.add_edge(START, "fetch_context")
graph.add_edge("fetch_context", "build_prompt")
graph.add_edge("build_prompt", "call_model")
graph.add_edge("call_model", END)
Output{"company_id": "acme", "context": "...", "prompt": "...", "raw_response": "{...model json...}"}
What it meansEach node adds one useful field. By the model call, the state contains everything the next step needs.

Project 1: content generation

Business content generation is already a graph

Start content request
  -> Fetch company, suburb, or province data
  -> Build structured SEO prompt
  -> Call OpenAI or Gemini provider
  -> Parse JSON response
  -> Repair or retry if broken
  -> Enforce expected schema
  -> Save generated content
  -> Update bulk job progress
  -> End
Graph view: every arrow can become a named node or route. If one step fails, you know exactly where the workflow stopped.
The model call is one step. The workflow around it is the real engineering.

Project 1 state

The content workflow state

from typing import TypedDict, Any

class ContentState(TypedDict, total=False):

    location_id: str
    location_type: str

What it meansState is the shared memory of one run. It carries request data, intermediate AI output, errors, retries, and the final saved result.

Project 1 nodes

Each service step becomes a node

load_contextadds company_info and location_info
build_promptadds prompt
call_ai_provideradds raw_ai_response

This replaces hidden control flow with named, inspectable steps.

Real failure path

Broken JSON gets its own node

import json
from typing import Any

def parse_json_node(state: ContentState) -> dict[str, Any]:
    # Keep parsing separate so failure can route cleanly.
    try:
        parsed = json.loads(state["raw_ai_response"])
        return {
            "parsed_content": parsed,
            "validation_errors": [],
        }
    except json.JSONDecodeError as error:
        return {
            "validation_errors": [f"json_error: {error}"],
        }
OutputValid JSON -> {"parsed_content": ..., "validation_errors": []} Broken JSON -> {"validation_errors": ["json_error: ..."]}
What it meansThe node records failure in state instead of hiding it. The next edge can now decide whether to retry, repair, or fail cleanly.

Conditional edge

Route invalid JSON to repair or failure

def route_after_parse(state: ContentState) -> str:
    has_errors = bool(state.get("validation_errors"))
    retries = state.get("retry_count", 0)

    if has_errors and retries < 2:
        return "repair_json"

    if has_errors:
        return "mark_failed"

    return "enforce_schema"
Outputretry_count 0 + errors -> repair_json retry_count 2 + errors -> mark_failed no errors -> enforce_schema
What it meansA conditional edge is just a small router function. It turns messy runtime situations into explicit workflow paths.
Conditional edges are just decisions based on state.

Project 1 routing

Add retry and save routes

graph.add_conditional_edges(
    "parse_json",
    route_after_parse,
    {
        "repair_json": "repair_json",
        "mark_failed": "mark_failed",
        "enforce_schema": "enforce_schema",
    },
)

graph.add_edge("repair_json", "call_ai")
graph.add_edge("enforce_schema", "save_content")
graph.add_edge("save_content", END)
graph.add_edge("mark_failed", END)

content_app = graph.compile()
OutputValid run -> save_content -> END Invalid run -> repair_json -> call_ai Repeated invalid run -> mark_failed -> END
What it meansThe retry loop is visible. Students can trace exactly why a page was saved, repaired, or marked failed.

Project 2: VARA licensing

Regulatory licensing is not one prompt

Client intake
  -> Required document readiness
  -> Document indexing and retrieval prep
  -> License activity classification
  -> Policy generation
  -> Operational diagram generation
  -> Regulatory mapping
  -> Gap analysis
  -> Human gap decisions
  -> Submission package export
  -> Feedback and remediation
Graph view: evidence, policy work, gaps, and human decisions should be separate steps so the workflow can be reviewed later.
This needs order, validation, evidence, auditability, and human approval.

Project 2 state

The licensing workflow state

from typing import TypedDict, Any

class LicensingCaseState(TypedDict, total=False):
    case_id: str
    applicant_profile: dict[str, Any]
    uploaded_documents: list[str]
    indexed_chunks: list[str]
    classification: dict[str, Any]
    required_policies: list[str]
    generated_policies: list[str]
    regulatory_mappings: list[dict[str, Any]]
    gap_findings: list[dict[str, Any]]
    human_decisions: list[dict[str, Any]]
    submission_package: str
    errors: list[str]
OutputAfter gap analysis: gap_findings is added. After review: human_decisions is added.
What it meansThe state separates facts, AI analysis, and human judgment. That makes a regulated workflow easier to audit.

Project 2 nodes

Each AI responsibility is bounded

Classification node

Reads profile and evidence. Writes activity, confidence, citations.

Mapping node

Reads policies and rule chunks. Writes rule mappings and rationales.

Gap node

Reads evidence and mappings. Writes severity and remediation actions.

Safety route

Do not continue without evidence

def route_after_classification(state: LicensingCaseState) -> str:
    result = state["classification"]

    has_evidence = bool(result.get("evidence"))
    confident = result.get("confidence", 0.0) >= 0.75

    if has_evidence and confident:
        return "generate_policies"

    return "human_review"
Outputevidence + confidence >= 0.75 -> generate_policies missing evidence or low confidence -> human_review
What it meansThe graph refuses to continue when classification is weak. This is the approval gate in code.
Unsupported AI output should not unlock the next regulated step.

Human-in-the-loop

Human decisions become graph state

def human_gap_review_node(state: LicensingCaseState) -> dict:
    high_risk = [
        gap for gap in state.get("gap_findings", [])
        if gap.get("severity") in {"high", "critical"}
    ]

    return {
        "review_required": len(high_risk) > 0,
        "open_high_risk_gaps": high_risk,
    }
Output{"review_required": true, "open_high_risk_gaps": [...]}
What it meansHuman review becomes graph state. The workflow can pause, resume, export, or route based on review status.
AI recommends. Humans own regulatory judgment.

LangSmith is the control room

Trace: one workflow run

Trace view: a LangSmith trace is a timeline of one run. It shows which node ran, what it received, what it returned, and where time/cost/errors happened.
LangGraph controls the path. LangSmith shows the evidence.

Tracing code

Trace important functions

from langsmith import traceable

@traceable(name="Generate SEO content")
def generate_content(prompt: str) -> str:
    return ai_provider.generate(prompt)

@traceable(name="Classify VARA activity")
def classify_activity(profile: dict, chunks: list[str]) -> dict:
    return classifier.invoke({
        "profile": profile,
        "evidence": chunks,
    })
OutputLangSmith trace names: - Generate SEO content - Classify VARA activity
What it meansThe function still runs normally, but LangSmith records it as a named trace/span so debugging is readable.
Good trace names make debugging readable.

# Docker Compose passes these into the container.

OutputContainer receives LANGSMITH_TRACING, LANGSMITH_API_KEY, and LANGSMITH_PROJECT.

Configuration enters through `.env`, not hardcoded graph nodes.

Quality checks

Evaluations make quality repeatable

def evaluate_generated_page(output: dict) -> dict:
    body = str(output).lower()
    return {
        "json_valid": isinstance(output, dict),
        "schema_match": "title" in output and "sections" in output,
        "mentions_location": "suburb" in body or "province" in body,
        "no_fake_best_claim": "best" not in body,
    }
Output{"json_valid": true, "schema_match": true, "mentions_location": true, "no_fake_best_claim": true}
What it meansAn evaluation converts “is this good?” into repeatable checks that can be compared across runs.
Eval examples: JSON validity, schema match, local relevance, no fake claims, grounded citations.

Real-world framing

Use real projects honestly

These projects are strong examples of workflows LangGraph can formalize. Do not claim an existing project uses LangGraph unless it actually does.

LangGraph would make the workflow structured and retryable. LangSmith would make it traceable and improvable.
