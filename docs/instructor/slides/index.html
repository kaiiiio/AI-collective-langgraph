What is an AI agent?

An AI agent is software that uses a model to decide what should happen next, then acts through tools, memory, or workflow steps.

Teach it this way: A chatbot replies. An agent pursues a goal. The difference is not magic; it is the loop of goal, context, decision, action, observation, and next decision.
  • Goal: what the user or system wants done.
  • State: what the agent knows right now.
  • Action: a tool call, node execution, or response.
  • Observation: the result that informs the next step.

Clear mental models

Agent vs chatbot vs workflow

Students often use “agent” for everything. Separate the words early so LangGraph has a reason to exist.

Chatbot

Input: user message

Path: usually one model call

Best for: simple Q&A and conversation

Workflow

Input: task data

Path: known sequence of steps

Best for: repeatable business processes

Agent

Input: goal + context

Path: decisions, tools, branches, memory

Best for: tasks where the next step depends on state

Key takeaway: LangGraph is useful when the path cannot be represented as one straight line.

Models need hands

Why agents need tools

A model can generate text. Tools let the system do reliable work outside the model.

Instructor cue: Ask students whether they want the model to guess an algebra answer, or call deterministic code that solves the equation. That is the tool boundary.
  • Calculate: exact math beats model guessing.
  • Fetch: APIs provide fresh external data.
  • Validate: deterministic code checks constraints.
  • Act: tools connect agents to real systems.

Same environment everywhere

Docker: why we use it

Docker keeps bootcamp setup boring. Everyone gets the same Python version, dependencies, and commands.

Without Docker, a workshop starts with “why is Python broken on my laptop?” With Docker, it starts with the lesson.
  • No local Python installation drama.
  • No virtual environment mismatch.
  • Fast reset between classes.

Two words students mix up

Image vs container

Use a cooking analogy: an image is the recipe plus packed ingredients; a container is the meal being cooked right now.

Image

Built from the Dockerfile.

Contains Python, dependencies, and project files.

Reusable and shareable.

Container

Created from an image.

Runs one command or service.

Disposable after the lesson finishes.

Key takeaway: build images; run containers.

Why LangGraph exists

The problem with simple chains

Chains are excellent until the next step depends on what happened earlier. Agents live in that “depends” zone.

ABC

Simple chain: always the same path. Easy to reason about, but limited when routing or memory matters.

RouterToolMemoryReview

Graph workflow: branches, pauses, loops, remembers, and exposes every transition.

Key takeaway: LangGraph gives structure to workflows that cannot stay linear.

The workflow engine

What is LangGraph?

LangGraph builds stateful AI workflows using nodes, edges, and typed state.

LangGraph does not replace the model. It organizes the system around the model so you can see, test, and control the workflow.
  • Nodes do work.
  • Edges define movement.
  • State carries context.
  • Routers choose branches.

Teach these five

LangGraph core concepts

These five concepts are the vocabulary students should repeat until they can explain every module.

Graph

The complete workflow.

Example: research -> writer -> reviewer.

State

The shared data object.

Example: message, route, response.

Node

A Python function.

Example: hello_node(state).

Edge

The connection between nodes.

Example: START -> hello -> END.

Router

A function that chooses.

Example: joke, fact, or tool.

The most important idea

State explained visually

State is the object passed between nodes. Each node reads current values and returns only the updates it owns.

Ask students: “What keys exist before this node runs? What keys changed after it returned?”

Edges are the map

Edges explained visually

Edges are the graph’s control flow. They say which node runs next after the current node finishes.

Agents need context

Memory and conversation history

Memory is how an agent carries useful context across turns. In this lab, memory starts as a simple message list so the idea is visible.

Keep memory concrete: it is not “the model remembering.” It is application state you store, retrieve, and pass back into the workflow.
  • Short-term memory: recent conversation history.
  • Working memory: current graph state.
  • Persistent memory: saved data across sessions.

From intent to action

Tool calling flow

Tool calling is a controlled handoff. The agent decides which tool is needed, the tool executes deterministic code, and the result returns to state.

Reason, act, observe

ReAct-style agent loop

ReAct is a teaching pattern for agent behavior: decide, act, observe, and continue until the answer is ready.

  • Reason: choose whether a tool is needed.
  • Act: execute the chosen tool or node.
  • Observe: store the result in state.
  • Answer: respond when enough context exists.

Roles as nodes

Multi-agent collaboration

Multi-agent does not have to mean chaos. In LangGraph, each “agent” can be a named node with a clear role and contract.

Research Agent

Input: question + documents.

Output: concise research summary.

Writer Agent

Input: summary.

Output: draft answer in user-facing language.

Reviewer Agent

Input: draft.

Output: approval, feedback, or revision request.

Key takeaway: agent collaboration is easier to debug when each role is just another graph node.

Enterprise control

Human-in-the-loop review

Human review lets automation pause before a risky or important action. This is common in enterprise workflows.

Use examples: publishing legal text, sending customer emails, approving policy decisions, or submitting a report.
  • Review draft output.
  • Approve or reject.
  • Publish only when safe.

Visibility for AI workflows

What is LangSmith?

LangSmith is observability for AI systems. It shows the graph run, each node, each tool call, and the data moving between them.

Traceone complete graph execution
router nodeinput: user_message / output: route
tool nodeinput: route / output: tool_result
answer nodeinput: state / output: response
Key takeaway: LangGraph runs the workflow; LangSmith shows what happened.

Observability vocabulary

Traces, spans, and runs

These terms help students read LangSmith screens without getting lost.

Trace

The full execution: START to END.

Span

A nested operation inside the trace.

Run

A recorded graph, node, model, or tool call.

Metrics

Latency, token usage, status, and errors.

Debug with evidence

Debugging graph execution

Instead of asking “why did the agent do that?”, inspect the state transition that caused the next branch.

  • Was the router input correct?
  • Did the node return the expected keys?
  • Was a tool called with the right argument?
  • Where did latency or errors appear?

How to build agents sanely

Best practices for agent architecture

Good agent architecture is mostly boring software engineering: clear boundaries, tests, and observability.

Typed state

Make workflow data explicit.

Students should know every key and who writes it.

Small nodes

Give each node one responsibility.

If a node is hard to name, split it.

Trace everything

Use LangSmith before guessing.

Debug state transitions, not vibes.

Key takeaway: the agent is only as understandable as its state, nodes, and traces.

Everything combined

Capstone: AI Research Assistant

The capstone combines research, writing, review, memory-shaped state, Docker, and LangSmith-ready execution.

Present this as composition, not magic. Students already saw every part in earlier modules.
  • Search-like document lookup.
  • Summarization into shared state.
  • Reviewer approval before final answer.
